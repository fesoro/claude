## TDD nədir, niyə lazımdır

Test Driven Development (TDD) - **əvvəlcə test yazıb, sonra kodu yazma** metodologiyasıdır. Kent Beck tərəfindən populyarlaşdırılmış bu yanaşmada inkişaf prosesi üç addımdan ibarətdir: test yaz (Red), testi keçir (Green), kodu təmizlə (Refactor).

**TDD niyə lazımdır:**

1. **Bug-ların erkən aşkarlanması** - Kod yazılmadan əvvəl test yazıldığı üçün bug-lar erkən tapılır
2. **Daha yaxşı dizayn** - Test yazmaq üçün kodun testable olması lazımdır, bu da daha yaxşı dizayna gətirib çıxarır
3. **Canlı sənədləşdirmə** - Testlər kodun necə işlədiyini göstərən canlı sənəddir
4. **Refactoring güvəni** - Mövcud testlər refactoring zamanı heç nəyin sınmadığına zəmanət verir
5. **Regression prevention** - Yeni dəyişikliklər köhnə funksionallığı pozmur
6. **Daha az debugging** - Testlər problemin harada olduğunu dəqiq göstərir

**TDD olmadan vs TDD ilə:**

```
TDD olmadan:
Kod yaz → Əl ilə test et → Bug tap → Düzəlt → Əl ilə test et → ...
(uzun, təkrarlanan, etibarsız proses)

TDD ilə:
Test yaz → Kod yaz → Test keçir → Refactor → ...
(qısa, avtomatlaşdırılmış, etibarlı proses)
```

## Red-Green-Refactor Cycle

TDD-nin əsas dövrəsi üç addımdan ibarətdir:

### 1. RED (Qırmızı) - Uğursuz test yaz

*1. RED (Qırmızı) - Uğursuz test yaz üçün kod nümunəsi:*
```php
// İlk addım: Hələ mövcud olmayan funksionallıq üçün test yaz
public function test_can_calculate_order_total_with_tax(): void
{
    $calculator = new OrderCalculator();

    $total = $calculator->calculateTotal(
        subtotal: 100.00,
        taxRate: 18.0, // 18% ƏDV
    );

    $this->assertEquals(118.00, $total);
}
// Bu test UĞURSUZ olacaq çünki OrderCalculator class-ı hələ yoxdur
```

### 2. GREEN (Yaşıl) - Testi keçirəcək minimum kodu yaz

*2. GREEN (Yaşıl) - Testi keçirəcək minimum kodu yaz üçün kod nümunəsi:*
```php
// İkinci addım: Testi keçirmək üçün ən sadə kodu yaz
class OrderCalculator
{
    public function calculateTotal(float $subtotal, float $taxRate): float
    {
        return $subtotal + ($subtotal * $taxRate / 100);
    }
}
// İndi test UĞURLU olacaq
```

### 3. REFACTOR (Yenidən qurmaq) - Kodu təmizlə

*3. REFACTOR (Yenidən qurmaq) - Kodu təmizlə üçün kod nümunəsi:*
```php
// Üçüncü addım: Kodu yaxşılaşdır, testlər keçməyə davam etməlidir
class OrderCalculator
{
    public function calculateTotal(float $subtotal, float $taxRate): Money
    {
        $tax = $this->calculateTax($subtotal, $taxRate);
        return Money::of($subtotal + $tax);
    }

    public function calculateTax(float $subtotal, float $taxRate): float
    {
        return round($subtotal * $taxRate / 100, 2);
    }
}

// Testi də yeniləyirik
public function test_can_calculate_order_total_with_tax(): void
{
    $calculator = new OrderCalculator();

    $total = $calculator->calculateTotal(
        subtotal: 100.00,
        taxRate: 18.0,
    );

    $this->assertEquals(Money::of(118.00), $total);
}

// Əlavə edge case testləri
public function test_can_calculate_total_with_zero_tax(): void
{
    $calculator = new OrderCalculator();

    $total = $calculator->calculateTotal(subtotal: 100.00, taxRate: 0);

    $this->assertEquals(Money::of(100.00), $total);
}

public function test_can_calculate_tax_separately(): void
{
    $calculator = new OrderCalculator();

    $tax = $calculator->calculateTax(subtotal: 100.00, taxRate: 18.0);

    $this->assertEquals(18.00, $tax);
}
```

**Dövrənin qızıl qaydaları:**
- Hər dəfə yalnız bir test əlavə edin
- Hər testi keçirmək üçün minimum kodu yazın
- Refactoring zamanı yeni funksionallıq əlavə etməyin
- Hər addımda testləri çalışdırın

## TDD vs BDD fərqi

### TDD (Test Driven Development)

- **Fokus:** Texniki implementasiya
- **Dil:** Developer terminologiyası
- **Alətlər:** PHPUnit, Pest
- **Yanaşma:** Unit test-lərdən başlayır

*- **Yanaşma:** Unit test-lərdən başlayır üçün kod nümunəsi:*
```php
// TDD üslubu - texniki, developer-oriented
class OrderCalculatorTest extends TestCase
{
    public function test_calculate_total_adds_tax_to_subtotal(): void
    {
        $calculator = new OrderCalculator();

        $result = $calculator->calculateTotal(100.00, 18.0);

        $this->assertEquals(118.00, $result);
    }

    public function test_throws_exception_for_negative_subtotal(): void
    {
        $calculator = new OrderCalculator();

        $this->expectException(InvalidArgumentException::class);

        $calculator->calculateTotal(-50.00, 18.0);
    }
}
```

### BDD (Behavior Driven Development)

- **Fokus:** Business davranışı
- **Dil:** Business terminologiyası (Given-When-Then)
- **Alətlər:** Behat, Pest (BDD syntax)
- **Yanaşma:** Feature/scenario-lardan başlayır

*- **Yanaşma:** Feature/scenario-lardan başlayır üçün kod nümunəsi:*
```php
// BDD üslubu - business-oriented, oxunaqlı
// Pest ilə BDD syntax
describe('Sifariş kalkulyatoru', function () {
    it('Subtotal-a ƏDV əlavə edərək ümumi məbləği hesablayır', function () {
        $calculator = new OrderCalculator();

        $result = $calculator->calculateTotal(100.00, 18.0);

        expect($result)->toBe(118.00);
    });

    it('Mənfi subtotal üçün xəta qaytarır', function () {
        $calculator = new OrderCalculator();

        expect(fn () => $calculator->calculateTotal(-50.00, 18.0))
            ->toThrow(InvalidArgumentException::class);
    });
});

// Behat ilə Gherkin syntax
// features/order_calculation.feature
/*
Feature: Sifariş hesablanması
    Müştəri kimi mən sifarişimin yekun məbləğini
    ƏDV daxil olmaqla görmək istəyirəm

    Scenario: ƏDV ilə sifariş hesablanması
        Given Subtotal 100.00 AZN-dir
        And ƏDV dərəcəsi 18%-dir
        When Yekun məbləğ hesablanır
        Then Nəticə 118.00 AZN olmalıdır

    Scenario: Endirimli sifariş
        Given Subtotal 200.00 AZN-dir
        And 10% endirim tətbiq olunur
        And ƏDV dərəcəsi 18%-dir
        When Yekun məbləğ hesablanır
        Then Nəticə 212.40 AZN olmalıdır
*/
```

## Test növləri

### Unit Test

Tək bir class və ya metodu test edir. Xarici asılılıqlar mock edilir.

*Tək bir class və ya metodu test edir. Xarici asılılıqlar mock edilir üçün kod nümunəsi:*
```php
// Bu kod unit test nümunəsini göstərir
class OrderCalculatorTest extends TestCase
{
    public function test_calculates_discount(): void
    {
        // Arrange
        $calculator = new OrderCalculator();

        // Act
        $discount = $calculator->applyDiscount(100.00, 10);

        // Assert
        $this->assertEquals(90.00, $discount);
    }

    public function test_does_not_allow_discount_over_100_percent(): void
    {
        $calculator = new OrderCalculator();

        $this->expectException(InvalidArgumentException::class);
        $this->expectExceptionMessage('Endirim 100%-dən çox ola bilməz');

        $calculator->applyDiscount(100.00, 150);
    }
}
```

### Integration Test

Bir neçə komponentin birlikdə işləməsini test edir (database, API çağırışları və s.).

*Bir neçə komponentin birlikdə işləməsini test edir (database, API çağı üçün kod nümunəsi:*
```php
// Bu kod inteqrasiya test nümunəsini göstərir
class OrderRepositoryTest extends TestCase
{
    use RefreshDatabase;

    public function test_can_create_and_retrieve_order(): void
    {
        // Arrange
        $user = User::factory()->create();
        $repository = new EloquentOrderRepository();

        // Act
        $order = $repository->create([
            'user_id' => $user->id,
            'status' => 'pending',
            'total' => 150.00,
        ]);

        $found = $repository->findById($order->id);

        // Assert
        $this->assertNotNull($found);
        $this->assertEquals($user->id, $found->user_id);
        $this->assertEquals(150.00, $found->total);
    }

    public function test_returns_null_for_nonexistent_order(): void
    {
        $repository = new EloquentOrderRepository();

        $result = $repository->findById(99999);

        $this->assertNull($result);
    }
}
```

### Feature Test

Tam HTTP request/response cycle-ını test edir.

*Tam HTTP request/response cycle-ını test edir üçün kod nümunəsi:*
```php
// Bu kod feature test nümunəsini göstərir
class OrderFeatureTest extends TestCase
{
    use RefreshDatabase;

    public function test_authenticated_user_can_create_order(): void
    {
        $user = User::factory()->create();
        $product = Product::factory()->create(['price' => 29.99, 'stock' => 10]);

        $response = $this->actingAs($user)
            ->postJson('/api/orders', [
                'items' => [
                    ['product_id' => $product->id, 'quantity' => 2],
                ],
                'shipping_address' => 'Bakı, Nəsimi rayonu',
            ]);

        $response->assertStatus(201)
            ->assertJson([
                'data' => [
                    'user_id' => $user->id,
                    'status' => 'pending',
                    'total' => 59.98,
                ],
            ]);

        $this->assertDatabaseHas('orders', [
            'user_id' => $user->id,
            'status' => 'pending',
        ]);
    }

    public function test_guest_cannot_create_order(): void
    {
        $response = $this->postJson('/api/orders', [
            'items' => [],
        ]);

        $response->assertStatus(401);
    }
}
```

### E2E (End-to-End) Test

Bütün sistemi (frontend + backend + database + xarici servislər) test edir.

*Bütün sistemi (frontend + backend + database + xarici servislər) test  üçün kod nümunəsi:*
```php
// Laravel Dusk ilə E2E test
class OrderE2ETest extends DuskTestCase

                ->assertSee('Sifariş təsdiqləndi')

        });
    }
}
```

## Test Pyramid

```
        /\
       /  \         E2E Tests (az)
      /    \        - Yavaş, bahalı
     /------\       - Tam sistem
    /        \
   /  Feature \     Integration Tests (orta)
  /   Tests    \    - Orta sürət
 /              \   - Komponentlər arası
/----------------\
/   Unit Tests    \ Unit Tests (çox)
/                  \ - Sürətli, ucuz
/                    \ - Tək komponent
```

**Test Pyramid qaydası:**
- **Unit Tests: ~70%** - Sürətli, çox, ucuz
- **Integration/Feature Tests: ~20%** - Orta sürət, orta miqdar
- **E2E Tests: ~10%** - Yavaş, az, bahalı

*- **E2E Tests: ~10%** - Yavaş, az, bahalı üçün kod nümunəsi:*
```php
// phpunit.xml-də test suite-ləri ayırmaq
/*
<testsuites>
    <testsuite name="Unit">
        <directory>tests/Unit</directory>
    </testsuite>
    <testsuite name="Feature">
        <directory>tests/Feature</directory>
    </testsuite>
    <testsuite name="Integration">
        <directory>tests/Integration</directory>
    </testsuite>
</testsuites>
*/
```

## PHPUnit əsasları

### Əsas Assertions

*Əsas Assertions üçün kod nümunəsi:*
```php
// Bu kod PHPUnit assertion metodlarının istifadəsini göstərir
class AssertionExamplesTest extends TestCase
{
    // Bərabərlik yoxlamaları
    public function test_equality_assertions(): void
    {
        $this->assertEquals(4, 2 + 2);          // Dəyər bərabərliyi (type coercion ilə)
        $this->assertSame(4, 2 + 2);            // Həm dəyər, həm tip bərabərliyi
        $this->assertNotEquals(5, 2 + 2);        // Bərabər deyil
        $this->assertTrue(true);                   // Boolean true
        $this->assertFalse(false);                 // Boolean false
        $this->assertNull(null);                   // Null
        $this->assertNotNull('value');             // Null deyil
    }

    // Array yoxlamaları
    public function test_array_assertions(): void
    {
        $array = ['name' => 'Orxan', 'age' => 25, 'city' => 'Bakı'];

        $this->assertArrayHasKey('name', $array);
        $this->assertArrayNotHasKey('email', $array);
        $this->assertCount(3, $array);
        $this->assertContains('Bakı', array_values($array));
        $this->assertEmpty([]);
        $this->assertNotEmpty($array);
    }

    // String yoxlamaları
    public function test_string_assertions(): void
    {
        $string = 'Laravel Framework';

        $this->assertStringContainsString('Laravel', $string);
        $this->assertStringNotContainsString('Symfony', $string);
        $this->assertStringStartsWith('Laravel', $string);
        $this->assertStringEndsWith('Framework', $string);
        $this->assertMatchesRegularExpression('/^Laravel/', $string);
    }

    // Exception yoxlamaları
    public function test_exception_assertions(): void
    {
        $this->expectException(InvalidArgumentException::class);
        $this->expectExceptionMessage('Dəyər boş ola bilməz');
        $this->expectExceptionCode(422);

        throw new InvalidArgumentException('Dəyər boş ola bilməz', 422);
    }

    // Instance yoxlamaları
    public function test_instance_assertions(): void
    {
        $order = new Order();

        $this->assertInstanceOf(Order::class, $order);
        $this->assertInstanceOf(Model::class, $order);
    }
}
```

## HTTP Tests

*HTTP Tests üçün kod nümunəsi:*
```php
// Bu kod HTTP endpoint-lərini test etməyi göstərir
class OrderApiTest extends TestCase
{
    use RefreshDatabase;

    // GET request
    public function test_can_get_order_details(): void
    {
        $user = User::factory()->create();
        $order = Order::factory()->for($user)->create();
        $items = OrderItem::factory()->count(3)->for($order)->create();

        $response = $this->actingAs($user)
            ->getJson("/api/orders/{$order->id}");

        $response->assertStatus(200)
            ->assertJson([
                'data' => [
                    'id' => $order->id,
                    'status' => $order->status,
                ],
            ])
            ->assertJsonCount(3, 'data.items');
    }

    // POST request
    public function test_can_create_order(): void
    {
        $user = User::factory()->create();
        $product = Product::factory()->create(['price' => 49.99, 'stock' => 10]);

        $response = $this->actingAs($user)
            ->postJson('/api/orders', [
                'items' => [
                    ['product_id' => $product->id, 'quantity' => 2],
                ],
                'shipping_address' => 'Bakı, Nizami küçəsi 10',
                'payment_method' => 'credit_card',
            ]);

        $response->assertStatus(201)
            ->assertJson([
                'data' => [
                    'user_id' => $user->id,
                    'status' => 'pending',
                    'total' => 99.98,
                ],
            ]);

        $this->assertDatabaseHas('orders', [
            'user_id' => $user->id,
            'total' => 99.98,
        ]);
    }

    // PUT request
    public function test_can_update_order_status(): void
    {
        $admin = User::factory()->admin()->create();
        $order = Order::factory()->create(['status' => 'pending']);

        $response = $this->actingAs($admin)
            ->putJson("/api/orders/{$order->id}", [
                'status' => 'confirmed',
            ]);

        $response->assertStatus(200)
            ->assertJson([
                'data' => [
                    'status' => 'confirmed',
                ],
            ]);
    }

    // DELETE request
    public function test_can_cancel_order(): void
    {
        $user = User::factory()->create();
        $order = Order::factory()->for($user)->create(['status' => 'pending']);

        $response = $this->actingAs($user)

{
    private ?User $user = null;
    private array $items = [];
    private string $status = 'pending';

    public static function anOrder(): self
    {

    {

        return $this;
    }

    public function withItem(Product $product, int $quantity = 1): self
    {
        $this->items[] = ['product' => $product, 'quantity' => $quantity];
        return $this;
    }

    public function withStatus(string $status): self
    {
        $this->status = $status;
        return $this;
    }

    {
        $this->couponCode = $code;
        return $this;
    }

    public function build(): Order
    {
        $user = $this->user ?? User::factory()->create();

        $order = Order::factory()->create([
            'user_id' => $user->id,
            'status' => $this->status,
        ]);

                'product_id' => $item['product']->id,
                'quantity' => $item['quantity'],
                'unit_price' => $item['product']->price,
            ]);
        }

        if ($this->couponCode) {
            $order->update(['coupon_code' => $this->couponCode]);
        }

        return $order->fresh(['items']);
    }
}

// Bu kod xəta idarəetməsini əlavə edərək GREEN mərhələyə keçişi göstərir
class PaymentProcessor
{
    public function __construct(
        private PaymentGatewayInterface $gateway,
        private LoggerInterface $logger,
    ) {}

    public function process(float $amount, string $currency, string $cardToken): PaymentResult
    {
        $this->validateAmount($amount);
        $this->validateCurrency($currency);

        try {
            $result = $this->gateway->charge($amount, $currency, $cardToken);

            $this->logger->info('Ödəniş uğurlu', [
                'amount' => $amount,
                'currency' => $currency,
                'transaction_id' => $result->transactionId,
            ]);

            return $result;
        } catch (PaymentGatewayException $e) {
            $this->logger->error('Ödəniş uğursuz', [
                'amount' => $amount,
                'currency' => $currency,
                'error' => $e->getMessage(),
            ]);

            return PaymentResult::failed(
                errorCode: 'gateway_error',
                errorMessage: $e->getMessage(),
            );
        }
    }

    private function validateAmount(float $amount): void
    {
        if ($amount <= 0) {
            throw new InvalidPaymentAmountException(
                "Ödəniş məbləği müsbət olmalıdır. Verilən: {$amount}"
            );
        }

        if ($amount > 50000) {
            throw new InvalidPaymentAmountException(
                "Ödəniş məbləği 50,000-dən çox ola bilməz. Verilən: {$amount}"
            );
        }
    }

    private function validateCurrency(string $currency): void
    {
        $supportedCurrencies = ['AZN', 'USD', 'EUR', 'TRY'];

        if (!in_array($currency, $supportedCurrencies)) {
            throw new UnsupportedCurrencyException(
                "Dəstəklənməyən valyuta: {$currency}"
            );
        }
    }
}
```

## İntervyu Sualları və Cavabları

### Sual 1: TDD nədir və Red-Green-Refactor cycle-ı izah edin.

**Cavab:** TDD əvvəlcə test yazıb, sonra kodu yazmaq metodologiyasıdır. Red-Green-Refactor cycle-ı üç addımdan ibarətdir: Red - hələ mövcud olmayan funksionallıq üçün uğursuz test yazılır. Green - testi keçirəcək minimum kod yazılır, mükəmməllik axtarılmır. Refactor - testlər keçdikdən sonra kod təmizlənir və yaxşılaşdırılır. Bu cycle təkrar olaraq davam edir. TDD-nin əsas faydası bug-ların erkən tapılması, daha yaxşı kod dizaynı və refactoring zamanı güvən yaratmasıdır.

### Sual 2: Unit Test ilə Integration Test arasında nə fərq var?

**Cavab:** Unit Test tək bir class və ya metodu izolə şəkildə test edir, xarici asılılıqlar mock edilir. Sürətli işləyir və ən kiçik kod vahidini yoxlayır. Integration Test isə bir neçə komponentin birlikdə işləməsini test edir, məsələn database ilə əlaqə, xarici API çağırışları və s. Daha yavaş işləyir amma real interaksiyanı test edir. Unit test-lər çox, integration test-lər orta sayda olmalıdır (Test Pyramid).

### Sual 3: Mock, Stub, Fake, Spy arasında nə fərq var?

**Cavab:** **Dummy** heç vaxt istifadə olunmur, yalnız parametr doldurmaq üçündür. **Stub** əvvəlcədən müəyyən edilmiş cavablar qaytarır, çağırışı yoxlamır. **Mock** həm cavab qaytarır, həm çağırışı yoxlayır (neçə dəfə, hansı parametrlərlə). **Spy** bütün çağırışları qeyd edir, test sonunda yoxlayırıq. **Fake** real implementasiyanın sadələşdirilmiş versiyasıdır, öz logic-i var, məsələn in-memory database.

### Sual 4: Laravel-də Event::fake(), Mail::fake() kimi fake-lər necə işləyir?

**Cavab:** Laravel-in fake mexanizmi facade-i əsl implementasiya əvəzinə fake implementasiya ilə əvəz edir. Məsələn, `Mail::fake()` real email göndərməz, əvəzinə bütün "göndərilən" email-ləri yaddaşda saxlayır. Sonra `Mail::assertSent()` ilə hansı email-lərin göndərildiyini yoxlaya bilərik. Bu, testləri sürətli edir və xarici servislərdən asılılığı aradan qaldırır. Hər fake öz assert metodlarını təklif edir.

### Sual 5: Code Coverage 100% testlərin yaxşı olduğu deməkdir?

**Cavab:** Xeyr. 100% code coverage kodun hər sətrinin ən az bir dəfə çalışdırıldığını göstərir, amma testlərin keyfiyyətini göstərmir. Test edge case-ləri, boundary conditions, error handling-i test etməyə bilər. Mutation testing bu problemi həll edir: kodda kiçik dəyişikliklər edib testlərin bu dəyişiklikləri tapıb-tapmadığını yoxlayır. 80-90% coverage əksər layihələr üçün yaxşı hədəfdir. Əsas olan coverage-in yüksəkliyi deyil, critical business logic-in tam test edilməsidir.

### Sual 6: TDD ilə BDD arasında nə fərq var?

**Cavab:** TDD texniki fokusludur, developer terminologiyası ilə unit test-lərdən başlayır. BDD isə business davranışına fokuslanır, Given-When-Then formatında yazılır və business stakeholder-ların anlaya biləcəyi dildə olur. TDD alətləri PHPUnit/Pest, BDD alətləri isə Behat kimi framework-lardır. TDD "bu metod düzgün işləyir?" sualına cavab verir, BDD isə "istifadəçi bu feature-u istifadə edəndə düzgün nəticə alır?" sualına cavab verir.

### Sual 7: Database Testing-də RefreshDatabase ilə DatabaseTransactions arasında fərq nədir?

**Cavab:** `RefreshDatabase` hər testdən əvvəl database-i migration-larla yenidən qurur. Daha yavaşdır amma tam təmiz database təmin edir. `DatabaseTransactions` isə hər testi database transaction-a sarır və test sonunda rollback edir, beləcə dəyişikliklər real olaraq commit olmur. Daha sürətlidir amma multi-database connection istifadə edəndə problem yarada bilər, çünki transaction yalnız bir connection-da işləyir.

### Sual 8: Pest PHP-nin PHPUnit-dən üstünlükləri nədir?

**Cavab:** Pest daha sadə və oxunaqlı syntax təklif edir: class yaratmaq lazım deyil, test/it/describe funksiyaları ilə testlər daha qısa yazılır. Expect API zəncirlənə bilən assertion-lar təklif edir. Dataset-lər DataProvider-lərin sadə alternatividir. Arch test-ləri kod arxitekturasını test etmək imkanı verir. Higher order test-lər bir sətirlik testlər yazmağa imkan verir. Pest PHPUnit üzərində qurulub, buna görə bütün PHPUnit özəllikləri ilə uyğundur.

### Sual 9: Real layihədə TDD tətbiq edərkən hansı çətinliklərə rast gəlirsiniz?

**Cavab:** Əsas çətinliklər: 1) Başlanğıcda daha yavaş inkişaf tempi - test yazmaq əlavə vaxt tələb edir, amma uzun müddətdə debug vaxtını azaldır. 2) Legacy kodda TDD tətbiq etmək çətin ola bilər - əvvəlcə refactoring lazım ola bilər. 3) UI/Frontend testləri üçün TDD çətin ola bilər. 4) Xarici API-lərlə inteqrasiya testləri yazmaq üçün yaxşı mock strategiyası lazımdır. 5) Komanda üzvlərinin TDD ilə tanış olması lazımdır.

### Sual 10: Hansı hallarda test yazmaq lazım deyil?

**Cavab:** Ümumiyyətlə bütün business logic test edilməlidir. Lakin bəzi hallarda test yazmaq əlavə fayda gətirmir: framework-un öz funksionallığını test etmək (Laravel-in validator-u artıq test edilib), sadə getter/setter metodları, dəyişmə ehtimalı olmayan konfiqurasiya faylları. Amma kritik business logic, API endpoint-ləri, complex hesablamalar, edge case-lər və security ilə bağlı funksionallıq mütləq test edilməlidir.

### Sual 11: Pest-in `arch()` testi nədir?

**Cavab:** Pest-in Architecture Testing xüsusiyyətidir. Kod arxitekturasının qaydalara uyğun olduğunu avtomatik yoxlayır. Nümunələr:
***Cavab:** Pest-in Architecture Testing xüsusiyyətidir. Kod arxitektur üçün kod nümunəsi:*

### Sual 12: Property-Based Testing nədir?

**Cavab:** Klassik testdə spesifik input dəyərləri yazırıq. Property-based testing isə test framework-ün özü minlərlə random input generate edir və müəyyən xassənin (property) həmişə doğru olduğunu yoxlayır. Məsələn: `Money::add(a, b)` — kommutatif olmalıdır: `a + b === b + a`. PHP-də `eris/eris` paketi, Pest-də experimental dəstək mövcuddur. Faydası: edge case-ləri özü tapır, bir sətir kod minlərlə test halını əhatə edir.

## Anti-patternlər

**1. Test-dən Sonra Kod Yazmaq (TTD Əvəzinə)**
Əvvəlcə kodu yazıb sonra test əlavə etmək — test artıq yazılmış kod üçün uyğunlaşdırılır, dizayn problemləri gec aşkarlanır, testability əvvəldən nəzərə alınmır. TDD-nin məqsədi testi əvvəl yazıb kod dizaynını test tərəfindən idarə etdirməkdir.

**2. Implementation Detail-lərini Test Etmək**
Private method-ları Reflection ilə, daxili state-i birbaşa test etmək — refactoring zamanı davranış dəyişmədən testlər sınır, tight coupling yaranır. Yalnız public interface-i test edin; private metodlar public davranış vasitəsilə örtülür.

**3. Həddindən Artıq Mock**
Hər dependency-ni mock edib sistemi real inteqrasiyadan tamamilə kəsmək — testlər keçir amma real mühitdə sistem sınır, mock-lar real davranışdan uzaqlaşır. Unit test-lər üçün mock-lar; integration test-lərdə real (ya da test database) inteqrasiya istifadə edin.

**4. Testləri Bir-birindən Asılı Etmək**
Bir testin yaratdığı dataya arxalanan başqa test yazmaq — test sırası dəyişdikdə ya da bir test uğursuz olduqda digərləri də sınır. Hər test müstəqil olmalıdır; `RefreshDatabase` ilə hər test üçün təmiz vəziyyət təmin edin.

**5. Yalnız "Happy Path" Test Etmək**
Yalnız uğurlu ssenarini test edib xəta hallarını, edge case-ləri, invalid input-ları unutmaq — kritik bug-lar məhz kənar hallarda yaranır. Hər feature üçün valid input, invalid input, boundary values və xəta hallarını test edin.

**6. Çox Geniş (Brittle) Test-lər Yazmaq**
HTTP response-un tam JSON strukturunu hardcode etmək, SQL sorğularının tam mətni ilə test etmək — kiçik UI/format dəyişiklikləri testləri sındırır, test dəyəri azalır. Yalnız davranışı doğrulayan assertion-lar yazın; əhəmiyyətsiz detalları assert etməyin.

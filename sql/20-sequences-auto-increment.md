# Sequences & AUTO_INCREMENT

> **Seviyye:** Beginner ⭐

Unique ID-lər avtomatik yaratmaq üçün. Primary key üçün ən çox istifadə olunur.

## AUTO_INCREMENT (MySQL)

```sql
-- Sadə
CREATE TABLE users (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(255)
);

INSERT INTO users (name) VALUES ('Ali');           -- id = 1
INSERT INTO users (name) VALUES ('Veli');          -- id = 2
INSERT INTO users (name) VALUES ('Orkhan');        -- id = 3

-- Başlangıc dəyər
CREATE TABLE users (...) AUTO_INCREMENT = 1000;

-- Sonradan dəyişdir
ALTER TABLE users AUTO_INCREMENT = 10000;

-- Hazırkı dəyər
SHOW TABLE STATUS LIKE 'users';
SELECT AUTO_INCREMENT FROM information_schema.TABLES 
WHERE TABLE_NAME = 'users';
```

## SERIAL / IDENTITY (PostgreSQL)

```sql
-- PostgreSQL SERIAL (köhnə, amma hələ istifadə olunur)
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255)
);

-- BIGSERIAL (BIGINT variantı, böyük table-lar üçün tövsiyə)
CREATE TABLE orders (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT
);

-- SMALLSERIAL - kiçik table-lar
CREATE TABLE categories (
    id SMALLSERIAL PRIMARY KEY,
    name VARCHAR(50)
);
```

**SERIAL nədir:** Əslində `INT + DEFAULT nextval('seq') + NOT NULL` shortcut-udur.

### IDENTITY (PG 10+, SQL Standard)

```sql
-- Daha müasir və SQL-standard
CREATE TABLE users (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name VARCHAR(255)
);

-- GENERATED BY DEFAULT - sən də əllə ID verə bilərsən
CREATE TABLE users (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
);

-- ALWAYS - manual ID qoyulmasına icazə verilmir
INSERT INTO users (id, name) VALUES (99, 'Ali');     -- ERROR GENERATED ALWAYS-lə
INSERT INTO users (name) VALUES ('Ali');             -- OK
```

**Tövsiyə:** Yeni table-lar üçün `IDENTITY` istifadə et, `SERIAL` legacy.

## CREATE SEQUENCE (PostgreSQL)

Sequence — ayrı obyekt, table-dan müstəqil.

```sql
-- Sequence yarat
CREATE SEQUENCE order_number_seq
    START WITH 10000
    INCREMENT BY 1
    MINVALUE 10000
    MAXVALUE 99999999
    NO CYCLE;                                  -- maxvalue-da error

-- Table-da istifadə et
CREATE TABLE orders (
    order_number INTEGER DEFAULT nextval('order_number_seq') PRIMARY KEY
);
```

### Sequence Funksiyaları

```sql
-- Növbəti dəyər (artırır)
SELECT nextval('order_number_seq');

-- Hazırkı dəyəri al (artırmır)
SELECT currval('order_number_seq');            -- yalnız bu session-da nextval çağırıldıqdan sonra

-- Spesifik dəyərə set
SELECT setval('order_number_seq', 50000);      -- növbəti nextval = 50001
SELECT setval('order_number_seq', 50000, false);   -- növbəti nextval = 50000

-- Sequence məlumatı
SELECT * FROM order_number_seq;
```

### CYCLE və ya Qaçınma

```sql
-- CYCLE - max-a çatanda min-ə qayıdır
CREATE SEQUENCE small_seq MAXVALUE 100 CYCLE;
-- 1, 2, ..., 100, 1, 2, ... (reuse!)

-- NO CYCLE (default) - max-dan sonra error
CREATE SEQUENCE strict_seq MAXVALUE 100 NO CYCLE;
```

**Qayda:** Primary key sequence-də CYCLE **istifadə etmə** — duplicate key yaradar.

## MySQL-də SEQUENCE Yoxdur

MySQL-də `AUTO_INCREMENT` var, ayrı sequence obyekti yox. Multiple counter lazımdırsa:

```sql
-- Trick: dummy table
CREATE TABLE invoice_seq (id INT AUTO_INCREMENT PRIMARY KEY);

-- Next value
INSERT INTO invoice_seq VALUES (NULL);
SELECT LAST_INSERT_ID();
DELETE FROM invoice_seq WHERE id < LAST_INSERT_ID();   -- clean up

-- Və ya Redis, application-level counter
```

## LAST_INSERT_ID() / RETURNING

### MySQL

```sql
INSERT INTO users (name) VALUES ('Ali');
SELECT LAST_INSERT_ID();                       -- 42

-- Session-scoped
-- Multi-row insert-də İLK row-un ID-i qaytarılır:
INSERT INTO users (name) VALUES ('A'), ('B'), ('C');
SELECT LAST_INSERT_ID();                       -- A-nın ID-i (məs 43)
-- B=44, C=45 olsa da, LAST_INSERT_ID() = 43
```

### PostgreSQL

```sql
-- RETURNING
INSERT INTO users (name) VALUES ('Ali') RETURNING id;

-- Sequence-dən əvvəlcədən al
SELECT nextval('users_id_seq');                -- 42
INSERT INTO users (id, name) VALUES (42, 'Ali');
-- Amma RETURNING daha yaxşıdır
```

## Sequence Gap-ləri

Auto-increment / sequence **qarantiyasız ardıcıllıq** verir. Gap-lər ola bilər:

```sql
-- User yarat
INSERT INTO users (name) VALUES ('Ali');       -- id = 1
INSERT INTO users (name) VALUES ('Veli');      -- id = 2
BEGIN;
INSERT INTO users (name) VALUES ('Bakir');     -- id = 3
ROLLBACK;                                       -- 3 "itdi" amma sequence 3-dən irəli
INSERT INTO users (name) VALUES ('Orkhan');    -- id = 4 (3 skip!)
```

**Sıra:** 1, 2, 4, ... (gap 3-də)

**Niyə:** Sequence transaction-a ROLLBACK edilmir. Yoxsa lock-lar tuta-tuta əcəl şəkildə işləyərdi.

**Qayda:** `id` sütunu **unique identifier**-dir, "row sayı" deyil. Gap normaldır!

## Unsigned AUTO_INCREMENT (MySQL)

```sql
-- Range-i artır (0-dan başlayır)
CREATE TABLE users (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(255)
);

-- BIGINT UNSIGNED: 0 to 18.4 × 10^18
-- BIGINT SIGNED:   -9.2 × 10^18 to 9.2 × 10^18
```

**Tövsiyə:** `BIGINT UNSIGNED` — mənfi ID olmaz, range ikiqat artır.

## Distributed Systems-də Sequence Problemi

Multiple DB instance olanda auto-increment çətinlik yaradır:

```
DB1:  id = 1, 2, 3, 4
DB2:  id = 1, 2, 3, 4    -- Duplicate!
```

**Həllər:**

### 1. AUTO_INCREMENT step (MySQL multi-master)

```sql
-- DB1: odd (1, 3, 5, ...)
SET auto_increment_increment = 2;
SET auto_increment_offset = 1;

-- DB2: even (2, 4, 6, ...)
SET auto_increment_increment = 2;
SET auto_increment_offset = 2;
```

### 2. UUID

```sql
-- PostgreSQL
CREATE TABLE users (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY
);

-- MySQL
CREATE TABLE users (
    id CHAR(36) DEFAULT (UUID()) PRIMARY KEY
);
```

### 3. ULID / Snowflake (sortable distributed ID)

Application-level generation. Ətraflı: `45-id-generation-strategies.md`.

## AUTO_INCREMENT Reset

```sql
-- MySQL
TRUNCATE TABLE users;                          -- counter 1-ə sıfırlanır
ALTER TABLE users AUTO_INCREMENT = 1;          -- əllə

-- PostgreSQL
TRUNCATE TABLE users RESTART IDENTITY;
ALTER SEQUENCE users_id_seq RESTART WITH 1;
```

**Xəbərdarlıq:** Production-da reset etmə - FK cəhətdən sınır, data sekvens pozur.

## Laravel Nümunəsi

```php
// Migration
Schema::create('users', function (Blueprint $table) {
    $table->id();                              // BIGINT UNSIGNED AUTO_INCREMENT PK
    $table->string('name');
});

// ID əvvəldən başla
Schema::create('orders', function (Blueprint $table) {
    $table->id()->from(10000);                 // 10000-dən başla
    // Və ya
    $table->bigIncrements('id')->from(10000);
});

// UUID PK
Schema::create('users', function (Blueprint $table) {
    $table->uuid('id')->primary();
    // Və ya:
    // $table->uuid('id')->default(DB::raw('gen_random_uuid()'))->primary();
});

// Model
class User extends Model {
    protected $keyType = 'string';              // UUID üçün
    public $incrementing = false;
    
    protected static function boot() {
        parent::boot();
        static::creating(function ($model) {
            $model->id = Str::uuid();
        });
    }
}

// Get last inserted ID
$user = User::create(['name' => 'Ali']);
echo $user->id;                                 // auto-filled

// Query Builder
$id = DB::table('users')->insertGetId(['name' => 'Ali']);
```

## Best Practices

1. **Primary key üçün `BIGINT` (MySQL) / `BIGINT IDENTITY` (PG)** — INT limit risklidir
2. **`UNSIGNED` (MySQL)** — range ikiqat
3. **Gap-ləri narahatlıq etmə** — normaldır
4. **Distributed systems üçün ULID və ya UUID** — auto-increment pozulur
5. **SERIAL yerinə IDENTITY (PG)** — yeni SQL standard
6. **AUTO_INCREMENT əlini əslən qoyma** — sequence-dən qiymət sonrakı üçün istifadə et

## Interview Sualları

**Q: AUTO_INCREMENT-də gap-lər niyə yaranır?**
A: Transaction ROLLBACK, INSERT IGNORE, UPDATE-dən sonra köhnə ID, multi-master replication. Sequence `rollback-safe` deyil (performans üçün). Gap normaldır.

**Q: AUTO_INCREMENT vs UUID — hansı daha yaxşıdır?**
A: 
- **AUTO_INCREMENT**: kompakt (8 bayt), sortable, oxunaqlı, amma sequential (enumeration risk), distributed-də problem
- **UUID**: distributed-friendly, predicting çətin, amma böyük (16 bayt), random-a görə B-Tree fragment, sort etmir
- **ULID/Snowflake**: hər ikisinin üstünlüyü — sortable + distributed

**Q: `LAST_INSERT_ID()` bir neçə row insert-də nə qaytarır?**
A: MySQL-də **ilk row-un ID**-ni. Qalanlar sequential-dir (`id, id+1, id+2, ...`) amma LAST_INSERT_ID yalnız başlanğıcı qaytarır.

**Q: `BIGINT` yoxsa `UUID` primary key?**
A: BIGINT — əgər single DB / monolith-də, performans critical-də. UUID — distributed / offline-first / public ID-dir (sequential ID-lər data eksplorasiyasına imkan verir). ULID kompromis.

**Q: PostgreSQL sequence transaction-da rollback olunur?**
A: **Xeyir**. Sequence nextval() rollback-safe deyil. Niyə: lock olmadan əməliyyat sürətli olur. Gap normaldır.
